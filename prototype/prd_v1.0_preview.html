一、售后模块统一入口

原型链接

客户端原型
1.1 需求背景

当前售后入口分散，用户需要从不同地方进入售后流程，体验不统一。需要建立统一的售后模块入口，整合所有售后相关功能。

1.2 需求目标
  • 更改现有订单进入售后板块逻辑，将已付款的订单全部添加至售后板块列表展示
  • 统一售后入口，用户可从个人中心直接进入售后模块
  • 展示用户所有售后单，按状态分类（处理中、已完成、已关闭）
  • 支持快速发起售后申请

售后单流转逻辑：订单已付款 → 进入售后申请Tab → 用户发起售后 → 生成售后单 → 进入处理中Tab → 售后完成/关闭 → 进入历史售后Tab

1.3 功能内容
字段名称功能说明业务规则
入口位置个人中心售后服务入口固定位置：个人中心 → 售后服务
入口图标售后/客服图标固定图标样式
入口显示显示处理中售后单数量红点数量=处理中Tab的售后单数量；无处理中售后单时不显示红点
1.3.1 Tab栏数据逻辑
Tab栏功能说明业务规则
售后申请展示用户可申请售后的订单列表订单已付款进入此Tab，无售后操作订单在此Tab保持不变；展示待发货、待收货、已完成订单
处理中展示进行中的售后单订单发起售后单后进入此状态，售后单未关闭和完成之前保持此状态不变；查看详情和填写物流单号功能沿用当前功能
历史售后展示已完成和已关闭的售后单展示已完成和已关闭的售后单；重新申请、查看详情沿用当前订单功能
1.4 影响端口
端口功能范围
客户端售后入口、售后列表、发起申请
二、售后入库信息展示

原型链接

商家端原型
2.1 需求背景

商家端售后详情页缺少入库信息展示，运营无法查看退货商品的入库状态和数量。

2.2 需求目标
  • 商家端售后详情页展示入库信息
  • 展示退货单号、应入数量、实入数量、良品数量、破损数量、入库状态
2.3 功能内容

售后入库信息列表（统一8个字段）：

字段名称字段说明数据来源
商品名称退货商品名称订单商品表
SKU规格商品SKU规格信息订单商品表
退货单号京东云仓返回的退货入库单号京东云仓接口：退货入库明细消息推送
应入数量订单商品数量订单商品表
实入数量京东云仓返回的实际入库数量京东云仓接口：退货入库明细消息推送
良品数量京东云仓返回的良品数量京东云仓接口：商品等级=100
破损数量京东云仓返回的残品/脏品数量京东云仓接口：商品等级=200（残品）+商品等级=400（脏品）
入库状态根据良品数量与应入数量对比判断系统计算：实入数量=0→未入库；良品=应入→已入库；良品<应入→少件
2.3.1 数量更新逻辑
字段名称更新时机更新规则
应入数量订单创建时从订单商品表取值，后续不变
退货单号京东云仓推送消息时从接口字段packageNo取值
实入数量京东云仓推送消息时从接口"实际入库数量"字段取值
良品数量京东云仓推送消息时统计商品等级=100的商品数量
破损数量京东云仓推送消息时统计商品等级=200（残品）+商品等级=400（脏品）的商品数量
入库状态良品数量更新时系统自动计算：对比良品数量与应入数量
2.3.2 操作按钮说明
按钮名称功能说明业务规则
确认入库确认退货商品已入库，继续售后流程保留当前功能，无改动
拒绝退货退款拒绝用户的退货退款申请保留当前功能，无改动

三、确认收货阻止

原型链接

客户端原型
3.1 需求背景

用户在存在未完成售后单的情况下点击确认收货，可能导致售后流程异常。需要在订单详情页增加售后提示和确认收货阻止逻辑。

3.2 需求目标
  • 订单详情页展示进行中售后单的提示信息
  • 用户存在未完成售后单时，阻止确认收货操作
  • 引导用户前往售后详情页处理售后
3.3 功能内容
字段名称功能说明业务规则
阻止条件判断是否阻止确认收货订单存在售后单，且售后单状态 ≠ 已完成 且 ≠ 已关闭
阻止行为隐藏确认收货按钮满足阻止条件时，隐藏"确认收货"按钮，不弹出提示
3.3.1 订单详情页提示卡片

字段名称字段说明业务规则
绿色提示卡片展示进行中售后单信息背景色#f0fff4，边框#c6f6d5；显示售后单号、售后类型、当前状态三个字段

四、订单补发

原型链接

商家端原型
4.1 需求背景

订单发货后可能出现漏发、错发、商品损坏等情况，需要商家对订单进行补发操作。当前系统不支持订单补发功能。

4.2 需求目标
  • 支持商家对待收货和已完成状态的订单发起补发
  • 补发不关联售后单，作为独立的发货流程
  • 支持一个订单多次补发，生成多个补发包裹
  • 支持删除补发包裹（仅限补发包裹）
4.3 功能内容
字段名称功能说明业务规则
补发入口订单列表/详情页显示"补发"按钮显示条件：订单状态 = 待收货 或 已完成
补发次数一个订单可多次补发无次数限制，支持无限补发
补发商品选择补发的商品SKU可补发任意在京东云仓存在的SKU，不限于原订单商品
补发数量输入补发商品数量可超过原订单数量，不进行数量校验
4.3.1 补发弹窗字段
字段名称字段说明业务规则
发货商品选择补发商品及数量下拉选择京东云仓存在的SKU；数量手动输入，支持数字，无上限校验
物流公司下拉选择物流公司选项固定，从系统物流公司列表获取
物流单号手动输入物流单号支持输入字母、数字组合，长度1-50字符
备注可选填备注信息支持输入中文、字母、数字，长度0-200字符，非必填
确认发货按钮提交补发信息点击后生成新的补发包裹，包裹列表新增一条记录
取消按钮取消补发操作点击后关闭弹窗，不保存任何信息
4.3.2 包裹管理
字段名称功能说明业务规则
包裹Tab订单详情物流Tab展示所有包裹包含原包裹和补发包裹
包裹标识区分原包裹和补发包裹原包裹无标识；补发包裹显示"补发"标签
包裹排序包裹列表排序规则按添加时间正序排列，越晚添加的在最后面
删除包裹删除补发包裹仅补发包裹可删除，原包裹不可删除；未发货、已发货状态均可删除；物理删除
修改物流修改包裹物流信息所有包裹均可修改物流公司和物流单号
4.4 影响端口
端口功能范围
商家端订单列表补发入口、补发弹窗、包裹管理
客户端订单详情查看所有包裹物流
五、物流填写提醒
5.1 需求背景

用户退货申请审核通过后，需要在规定时间内填写退货物流信息。部分用户忘记填写导致售后超时关闭。

5.2 需求目标
  • 客户端：提醒用户及时填写退货物流，超时自动关闭
  • 商家端：提醒运营及时填写发货单号完成售后
5.3 功能内容
5.3.1 客户端提醒
字段名称功能说明业务规则
倒计时卡片显示售后关闭倒计时显示"距离售后关闭还剩 X天 X小时"
提醒渠道多渠道提醒用户微信小程序订阅消息 + 短信通知
首次提醒审核通过后提醒审核通过后24小时内发送
超时提醒临近超时前提醒临近超时前1天发送
超时处理超时未填写处理超时未填写自动关闭售后申请
超时时间计算计算剩余时间审核通过时间 + 7天 = 超时关闭时间；剩余时间 = 超时关闭时间 - 当前时间

六、重置售后期

原型链接

商家端原型
6.1 需求背景

部分订单已过售后申请时效，但用户有合理售后需求，需要运营手动重置售后期。

6.2 需求目标
  • 运营可对已过期订单重置售后期
  • 重置后用户可在新的售后期内申请售后
  • 记录操作人和操作时间
6.3 售后超期逻辑
字段名称功能说明业务规则
售后期计算计算订单售后期限订单完成时间 + 7天 = 售后期截止时间
超期判断判断订单是否已过售后时效当前时间 > 售后期截止时间 → 已超期；当前时间 ≤ 售后期截止时间 → 未超期
6.4 功能内容
6.4.1 重置售后期按钮
字段名称功能说明业务规则
按钮名称操作按钮显示文字固定显示"重置售后期"
显示位置按钮展示位置商家端订单列表/订单详情页操作列
显示条件按钮显示逻辑同时满足：①订单状态 = 已完成；②当前时间 > 售后期截止时间（已超期）
隐藏条件按钮隐藏逻辑①订单状态 ≠ 已完成；②当前时间 ≤ 售后期截止时间（未超期）；③已点击重置售后期且新的售后期未过期
6.4.2 重置售后期业务逻辑
字段名称功能说明业务规则
点击操作点击按钮后的处理弹出二次确认弹窗，文案："确定重置该订单的售后期吗？重置后用户将可在7天内申请售后"
确认重置确认后的业务逻辑以当前时间为起点，重新计算售后期：当前时间 + 7天 = 新的售后期截止时间；订单状态保持"已完成"不变
重置后效果重置售后期后的变化①客户端：该订单重新显示售后申请入口；②商家端："重置售后期"按钮隐藏
记录日志操作记录记录操作人、操作时间、原售后期截止时间、新售后期截止时间
6.5 影响端口
端口功能范围
商家端订单列表重置售后期按钮
客户端重置售后期后显示售后入口
系统售后期计算、记录操作日志
